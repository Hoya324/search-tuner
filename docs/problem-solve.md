# Problem Solve — Korean Search Tuner

## 1. 가장 어려울 것으로 예상되는 부분

### 동의어 자동 생성의 "품질" 문제

이게 프로젝트에서 가장 핵심이면서 가장 어려운 부분이다. 단순히 LLM한테 "동의어 만들어줘"라고 하면 쓸 수 없는 결과가 나온다.

**문제 1 — 다의어 충돌**: "배"라는 단어가 있으면 LLM이 과일 배, 선박 배, 신체 배를 전부 동의어로 묶어버릴 수 있다. 커머스에서 "배"를 검색하면 과일도 나오고 배낭도 나오는 참사가 발생한다.

**문제 2 — 동의어 범위 판단**: "패딩"과 "다운자켓"은 동의어인가? 엄밀히는 패딩은 충전재 종류에 무관한 상위 개념이고, 다운자켓은 오리/거위털 충전 제품이다. 커머스에서는 동의어로 취급하는 게 맞지만, LLM이 이런 도메인 특수성을 일관되게 판단하기 어렵다.

**문제 3 — co-occurrence 추출의 한계**: 상품 데이터에서 "캐나다구스"와 "캐구"가 함께 등장하는 경우가 거의 없을 수 있다. 같은 상품을 다른 이름으로 부르는 거니까 한 상품에 두 표현이 동시에 있을 이유가 없다. 그래서 단순 co-occurrence만으로는 동의어 후보를 못 뽑고, 다른 전략이 필요하다.

**대안 접근**: 검색 로그(사용자가 "캐구" 검색 → 클릭한 상품이 "캐나다구스 패딩")를 활용하는 게 가장 강력하지만, 오픈소스 데모에서는 검색 로그가 없으므로 **LLM의 세계 지식에 의존**하는 방향으로 간다. 즉 "한국 커머스에서 이 상품명들의 동의어/줄임말을 찾아줘" 방식.

### Nori 분석기 "자동 추천"의 평가 기준

분석기 설정을 3개 만들어서 비교하는 건 쉬운데, **어떤 게 "더 좋은" 토큰화인지 판단하는 기준**이 모호하다. "지리산"이 "지리"+"산"으로 나뉘는 게 좋은 건지 나쁜 건지는 검색 의도에 따라 달라진다. 이걸 LLM이 판단하려면 프롬프트 설계가 까다롭고, 결국 검색 품질 평가(Module 4)와 엮어야 제대로 된 추천이 가능하다.

### LLM-as-a-Judge의 신뢰도

LLM이 검색 결과의 relevance를 채점하는데, 이 채점 자체가 얼마나 믿을 수 있는지가 문제다. 논문들에 따르면 GPT-4 수준에서 사람과 80% 정도 일치하는데, 한국어 커머스 도메인에서는 그보다 낮을 수 있다. 특히 "무선 블루투스 이어폰"과 "에어팟 프로 2세대"가 얼마나 관련 있는지 같은 판단은 도메인 지식이 필요하다.

## 2. 기술적 난관

### ES 동의어 사전 동적 업데이트

동의어 사전을 바꾸려면 인덱스를 close → 설정 변경 → open 해야 하는데, 이 동안 검색이 안 된다. `updateable: true` 옵션으로 search analyzer의 동의어를 reload할 수 있긴 한데, ES 버전에 따라 동작이 다르고 제약이 있다. 이 부분을 깔끔하게 처리하는 게 생각보다 까다롭다.

### 샘플 데이터 품질

한국어 상품 데이터 10,000건을 직접 만들어야 하는데, 현실적인 데이터를 만드는 게 꽤 노동 집약적이다. LLM으로 생성하면 패턴이 단조로워지고, 실제 커머스 데이터의 "지저분함"(오타, 영한 혼용, 특수문자)을 재현하기 어렵다.

### Spring AI 성숙도

Spring AI가 GA 직전까지 API가 바뀌었다. 특히 structured output parsing(JSON 응답 강제)이 Provider마다 동작이 달라서, 프롬프트에서 "반드시 JSON으로 응답해"라고 해도 가끔 마크다운으로 감싸서 오거나 설명을 덧붙인다.

## 3. LLM 프롬프트 설계

### 3.1 동의어 생성 프롬프트

```
[System]
당신은 한국 이커머스 검색 엔진의 동의어 사전 전문가입니다.
주어진 상품명 목록을 분석하여 동의어 그룹을 생성합니다.

규칙:
1. 동의어는 "같은 상품/개념을 다르게 표현한 것"만 포함합니다.
2. 상위 개념-하위 개념 관계는 동의어가 아닙니다. (예: 신발 ≠ 운동화)
3. 브랜드명의 한글/영문/줄임말 변형은 동의어입니다. (예: 나이키 = Nike = NIKE)
4. 한국 소비자들이 실제로 사용하는 줄임말을 포함합니다. (예: 캐나다구스 = 캐구)
5. 다의어 주의: "배"는 과일/선박/신체 중 카테고리 컨텍스트에 맞는 의미만 동의어로 묶습니다.
6. confidence가 0.7 미만인 불확실한 동의어는 제외합니다.

출력 형식 (JSON만, 다른 텍스트 없이):
{
  "synonymGroups": [
    {
      "terms": ["term1", "term2", "term3"],
      "type": "EQUIVALENT",
      "confidence": 0.95,
      "reasoning": "한 줄 근거"
    }
  ]
}

[User]
카테고리: {category}
상품명 목록 (총 {count}개):
{product_names}

이 상품명들에서 동의어 그룹을 추출해주세요.
```

**핵심 설계 포인트:**

- "상위-하위 개념은 동의어 아님"을 명시하지 않으면 LLM이 "신발→운동화→나이키 운동화"를 전부 동의어로 묶는다
- "다의어 주의"를 명시하지 않으면 "배" 문제가 발생한다
- confidence threshold를 지정하지 않으면 억지 동의어가 대량 생성된다
- reasoning 필드를 넣으면 LLM이 더 신중하게 판단하고, 사람 리뷰 시에도 도움이 된다

**예상 문제와 대응:**

| 문제 | 발생 빈도 | 대응 |
|------|----------|------|
| JSON 포맷 깨짐 | ~10% | response에서 ```json 태그 strip + JSON repair 라이브러리 |
| 과도한 동의어 그룹핑 | ~20% | confidence threshold 0.7로 필터링 + 사람 리뷰 단계 |
| 한국어 줄임말 누락 | ~30% | 프롬프트에 줄임말 예시를 few-shot으로 추가 |
| 입력 토큰 한도 초과 | 상품 2000개+ | 배치로 나눠서 호출 (500개씩) → 결과 merge |

### 3.2 분석기 추천 프롬프트

```
[System]
당신은 Elasticsearch Nori 한글 형태소 분석기 전문가입니다.
여러 분석기 설정으로 동일한 텍스트를 토큰화한 결과를 비교하고,
주어진 도메인에 가장 적합한 설정을 추천합니다.

평가 기준:
1. 검색 재현율 (recall): 사용자가 다양한 표현으로 검색해도 원하는 상품을 찾을 수 있는가?
2. 검색 정밀도 (precision): 불필요한 토큰이 매칭되어 엉뚱한 결과가 나오지 않는가?
3. 복합명사 처리: "무선블루투스이어폰"을 적절히 분리하는가?
4. 고유명사 보존: "캐나다구스", "지리산" 같은 고유명사를 잘못 분리하지 않는가?

출력 형식 (JSON만):
{
  "recommendation": "config_name",
  "reasoning": "추천 근거 2~3문장",
  "tradeoffs": "이 설정의 단점/주의사항",
  "configDetails": { ... ES settings JSON ... }
}

[User]
도메인: {domain} (예: 패션 커머스)
테스트 텍스트: {sample_texts}

토큰화 비교 결과:
- Config A (decompound=none): {tokens_a}
- Config B (decompound=discard): {tokens_b}
- Config C (decompound=mixed): {tokens_c}

어떤 설정이 이 도메인에 가장 적합합니까?
```

### 3.3 Relevance 판단 프롬프트 (LLM-as-a-Judge)

```
[System]
당신은 이커머스 검색 결과의 관련성을 평가하는 전문가입니다.
사용자의 검색 쿼리와 검색된 상품이 얼마나 관련 있는지 0~3점으로 평가합니다.

점수 기준:
- 3 (Perfectly relevant): 검색 의도에 정확히 부합하는 상품
- 2 (Highly relevant): 검색 의도와 관련이 높지만 완벽하지는 않은 상품
- 1 (Partially relevant): 간접적으로 관련이 있는 상품
- 0 (Irrelevant): 검색 의도와 무관한 상품

예시:
- 쿼리 "캐구 패딩", 상품 "캐나다구스 남성 롱패딩" → 3점
- 쿼리 "캐구 패딩", 상품 "노스페이스 숏패딩" → 1점
- 쿼리 "캐구 패딩", 상품 "여성 캐시미어 코트" → 0점

출력 형식 (JSON만):
{ "score": 3, "reasoning": "한 줄 근거" }

[User]
검색 쿼리: "{query}"
상품 정보:
- 상품명: {product_name}
- 카테고리: {category}
- 브랜드: {brand}
- 설명: {description}

이 상품은 검색 쿼리에 얼마나 관련이 있습니까?
```

**핵심 설계 포인트:**

- few-shot example을 넣어야 채점 기준이 일관된다. 없으면 LLM이 매번 다른 기준으로 채점한다
- 상품의 카테고리, 브랜드 정보를 함께 줘야 판단 정확도가 올라간다
- reasoning을 요구하면 Chain-of-Thought 효과로 판단 품질이 향상된다

## 4. LLM 비용 추정

| 작업 | 호출 수 | 예상 토큰 | GPT-4o 비용 | GPT-4o-mini 비용 |
|------|---------|----------|------------|----------------|
| 동의어 생성 (상품 2,000개) | ~4회 (500개씩 배치) | ~40,000 토큰 | ~$0.60 | ~$0.06 |
| 분석기 추천 (3개 설정 비교) | ~1회 | ~3,000 토큰 | ~$0.05 | ~$0.005 |
| 품질 평가 (100 쿼리 × top-10) | ~1,000회 | ~500,000 토큰 | ~$7.50 | ~$0.75 |
| A/B 비교 (×2) | ~2,000회 | ~1,000,000 토큰 | ~$15.00 | ~$1.50 |

품질 평가가 비용이 가장 크므로, **개발/데모 시에는 GPT-4o-mini를 쓰고, 실제 평가 시에만 GPT-4o를 쓰는** 전략이 현실적이다. 프롬프트에서 이걸 설정으로 분리해두면 된다.

## 5. 위험 요소 & 대응 전략 요약

| Risk | Impact | Likelihood | Mitigation |
|------|--------|-----------|------------|
| LLM 동의어 품질 불안정 | 높음 | 높음 | confidence threshold + 사람 리뷰 UI + few-shot 예시 강화 |
| JSON 응답 파싱 실패 | 중간 | 중간 | JSON repair 라이브러리 + retry 로직 + fallback 파서 |
| ES 동의어 동적 업데이트 제약 | 중간 | 높음 | search_analyzer만 updateable로 설정, 필요시 reindex API 제공 |
| 샘플 데이터 비현실성 | 낮음 | 높음 | 공공 데이터 활용 + LLM 생성 + 수동 보정 혼합 |
| Spring AI API 변경 | 중간 | 중간 | Adapter 패턴으로 격리, Spring AI 버전 고정 |
| LLM 비용 초과 | 중간 | 낮음 | 모델별 설정 분리 (mini/full), 배치 크기 조절, 캐싱 |
| 한국어 특화 평가 부정확 | 중간 | 중간 | 프롬프트에 한국어 커머스 few-shot 예시 5~10개 포함 |

가장 큰 리스크는 **LLM이 생성한 동의어를 그대로 적용하면 검색이 망가질 수 있다**는 것이다. 그래서 반드시 "생성 → 사람 리뷰 → 적용" 워크플로우를 기본으로 가고, 자동 적용은 옵션으로 제공하는 게 안전하다.
